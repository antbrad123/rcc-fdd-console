One clear way of setting up fault detection for any building we monitor

— what we detect, how we build it, and how we measure coverage

1. What This Document Is For

This is the RCC playbook for fault detection (FDD). It answers three questions: what we commit to detecting on each type of equipment, how we design and build the detection logic for each individual building, and how we measure whether a building is properly covered. It applies to every building we onboard, whatever the client, BMS make or age of the controls.

How to read it: Sections 2–3 explain the thinking. Sections 4–5 are the working rules (naming, severities, sign-off stages). Section 6 is the step-by-step onboarding process. Section 9 shows it all worked through for AHUs. If you only read one thing, read Section 6 and the AHU table.

Why we need it: every building arrives different — different contractors, different point names, different sensors fitted. Without a shared playbook, what “good FDD” means gets re-argued for every site and no two buildings can be compared. With it, every building goes through the same steps and reports the same two numbers: data readiness and FDD coverage.

Covers: HVAC plant and terminal units, water and electrical infrastructure, energy/metering checks and BMS health monitoring, built in KODE OS as FDD Routines and Events. We also monitor the health and status of life-safety plant (fire alarm interface, smoke control, generators, ACB/ATS) via BMS interfaces — this supplements, and never replaces, the regulated response chains. Doesn’t cover (yet): lifts, security systems.

2. The Six Things We Hold To

  • Same process everywhere, logic built per building. We fix the list of faults to detect and the way we work. The actual routine logic is built fresh for each building, around its real points and control sequences. Same outcomes, bespoke builds.
  • No debates about scope. What a unit “should” have is read straight off the fault table using a few yes/no questions about the unit. Two people looking at the same unit get the same answer.
  • Build to the building, write it down. Designing around each building’s quirks is the point — but every routine gets a short design record, and anything we can’t build (missing sensor, missing point) gets logged as Blocked. Nothing quietly disappears.
  • Coverage means coverage of what applies — measured two ways. Live coverage reflects FDD concepts that are active in KODE through an enabled or validated routine. Validated coverage is tracked separately and represents routines that have completed the tuning and assurance period. Both are measured against the asset’s own applicable set, and Blocked items show up in the numbers — never hidden.
  • A noisy alert is worse than no alert. Enabled gets a routine into live coverage, but only Validated — tuned, trusted, acted on — counts as assured. If site teams stop believing the alerts, the whole system is dead weight.
  • Alerts are written for the person receiving them. Anyone should understand an alert from the notification alone — “3PSQ – AHU-03 – Supply Fan Failed – Critical”. The engineering detail lives in tags behind the scenes.

3. The Fault Tables: What Applies to What

Each equipment type gets one table: the list of faults we commit to detecting. Each row describes the symptom, how serious it is, and roughly how to detect it — not the exact logic. Each row also says when it applies: “Always”, or only if the unit has a certain feature (“Has heating coil”, “VFD-driven”). So working out what applies to any unit is just answering a few yes/no questions. How each fault actually gets detected is then designed per building.

This is what makes coverage fair: a basic extract AHU and a complex fresh-air unit are both at 100% when everything that applies to them is live. Nobody is penalised for having simple kit, and nobody gets credit for ignoring complex kit.

The Operator Selection Tool applies this automatically. An operator or designer selects the asset class, ticks the features the asset has, and the tool returns the required FDD references with criticality and OOH routing. That returned list is the coverage denominator for the asset — it goes into the register and the design record, so scope is derived, not debated.

Events count too — through the same lifecycle. A fault concept may be detected by a KODE Event (instantaneous, unambiguous conditions such as trip signals or comms loss) or by an FDD Routine (anything needing persistence or multi-point logic). Both follow the same lifecycle: an FDD satisfied by an existing Event is marked Enabled while live, and Validated once confirmed suitable. There is one lifecycle for everything; Validated is the only fully assured state.

What’s flexible and what isn’t. Flexible: the detection logic itself — build it however the building’s points and sequences demand. Fixed: what gets detected, the severity words, the alert naming, thresholds as adjustable parameters (never hard-coded), and the short design record. If a building simply doesn’t have the data for a fault, it’s logged as Blocked so it stays visible.

4. Naming Alerts, Severities & Thresholds

4.1 How alerts are named

Routine names land word-for-word in people’s notifications, so they all follow one pattern:

[Building] – [Unit] – [Symptom] – [Severity]

Examples: “3PSQ – AHU-03 – Supply Fan Failed – Critical”; “T2 – Chiller 1 – Not Achieving Flow Temp – Warning”.

  • Name the unit, not the class. Use the identifier site teams use day-to-day (AHU-03, not “AHU”). The alert must be actionable without opening the platform.
  • Phrase the symptom, not the logic. “Supply Fan Failed”, not “Cmd/Status Mismatch”.
  • Engineering identity lives in tags. Every routine is tagged with the fault concept ID it implements (e.g. AHU-01) and the building design pack version. Names are for humans; tags are how building-specific routines reconcile to the central concept table and feed the coverage dashboard.

4.2 The four severity words (no others, ever)

SeverityDefinitionExpected response
CriticalLoss of service, safety-adjacent issue, statutory risk, or imminent asset/client impact.Immediate RCC triage; call-out path where OOH-flagged; stakeholder notification.
WarningService degradation, comfort risk, abnormal operation, or fault likely to require engineering review.Same/next business day investigation; work order raised on confirmation.
AlertEfficiency issue, early failure indicator, optimisation opportunity, or sustained non-critical abnormal behaviour.Batched review; feeds maintenance planning and energy reporting.
InfoVisibility, policy, or low-risk monitoring item.Periodic review; no action urgency.

OOH is a routing flag, not a severity: OOH-flagged FDDs also route to the out-of-hours call-out chain. An OOH fault is almost always Critical, but not every Critical is OOH.

4.3 Thresholds and tuning

  • Never hard-code a threshold. Every adjustable number lives in a KODE Parameter block, so tuning never means rewiring logic.
  • Every parameter gets a default and a plain-English Hint. e.g. “Time allowed for the fan to ramp up before we expect proof — raise for soft-start plant.” Anyone tuning it later should understand it instantly.
  • The more serious the fault, the faster we flag it. Rough defaults: Critical within 2–5 minutes; Warning 30–60 minutes; Alert hours to days; Info periodic. (In KODE: Buffer Time feeding an Array block — e.g. Array Min for “true the whole time”.)
  • One failure = one alert. Routines stay quiet during startup and maintenance, and a major fault mutes the knock-on ones — a dead fan shouldn’t also trigger five temperature alarms on the same unit.

5. The Four Stages of a Routine (and Blocked)

Every routine is in exactly one of these states. Enabled and Validated both count as live coverage; only Validated counts as assured — “enabled” isn’t “done”.

7. The Numbers Every Building Reports

From day one, each building carries three simple percentages:

  • Data readiness % — of the data this building’s fault list needs, how much can we actually get? A low score is an integration/points job, not an FDD job.
  • Live coverage % — of the in-scope FDDs, how many are active in KODE through an Enabled or Validated routine? This is the operational headline.
  • Validated coverage % — of the in-scope FDDs, how many have completed the tuning and assurance period? This is the assured figure, tracked separately and always at or below live coverage.

Behind those sit the working views: a portfolio heatmap by equipment type, the Backlog→Built→Configured→Enabled→Validated funnel, the criticality grid, and the blocked register (what’s blocked, why, and who owns fixing it). Together they turn “why isn’t FDD done?” into “building X needs two days of point mapping to unlock 40 routines.”

How the dashboard works. The RCC Coverage Dashboard reads the estate register file locally in the operator’s browser — nothing is uploaded. Once connected, it remembers the register and refreshes automatically when the file is saved (supported browsers), with a Refresh option and a visible last-updated time elsewhere. If the browser asks for permission again after a restart, one click reconnects.

8. Keeping It Under Control

  • The fault tables are owned centrally; logic is owned per building. Add a fault to the central table and every building’s to-do list updates automatically. Change a building’s logic, and it’s recorded in that building’s own design pack.
  • A second pair of eyes on every design. Because logic is bespoke, quality comes from review: every routine design gets peer-checked before it’s built. Clever solutions from one building get written into the table’s guidance for the next.
  • In KODE OS terms: faults are built as FDD Routines ending in an Alarm Trigger, thresholds as Parameter blocks, time-persistence as Buffer + Array blocks, multi-stage plant via Read Arrays, and every routine is tagged with its fault ID. Routine Export snapshots each building’s routines as a backup record; Batch Configuration helps with bulk parameter changes.
  • New buildings arrive ready. The fault tables and their data requirements go to contractors as part of the spec for new buildings and BMS replacements — so the data-gap problem stops growing.

9.3 Designer notes (common variations)

  • Fan proof (AHU-01/02): design to whatever proof the building offers — current switch, DP switch or VFD feedback — and write the diagnosis text to match (DP proof: add belt/switch checks; VFD feedback: note belt/coupling failure not excluded). No proof point at all → concept Blocked; inference-only detection is not permitted for Critical concepts.
  • Economizer concepts (AHU-16/17): use dry-bulb comparison where that is what the building has; design the enthalpy comparison where humidity sensing exists. Record the chosen approach in the design pack.

9.4 Two units, side by side

A toilet-extract AHU (no coils, fixed speed, no economizer, DP proof) answers its flags and reads off concepts AHU-01 to 08 — 8 applicable. A full fresh-air unit with both coils, economizer, VFD, static and filter sensors reads off all 24. Each then receives building-specific routine designs for its applicable concepts, and both report coverage against their own set.
